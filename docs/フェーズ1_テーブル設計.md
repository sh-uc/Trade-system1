# フェーズ1 テーブル設計（初版）

この文書は、フェーズ1で必要になる
- 推奨パラメータ管理
- 建玉管理
- 約定履歴
の3つのテーブル設計をまとめた初版です。

目的は、`backtests_runs` / `backtests_trades` で得た知見を、日次の実運用シミュレーションへつなげることです。

---

## 1. 設計方針

### 1.1 今回の設計でやりたいこと

- 銘柄ごとの推奨パラメータセットを正式に保持する
- 日次シミュレーション時に、どのパラメータで売買したか追えるようにする
- 建玉ごとの状態を保持できるようにする
- 売買が発生した履歴を、後から損益計算できる形で残す

### 1.2 今回は入れないもの

- 本番発注API連携
- 信用区分や空売り管理
- 複数口座管理
- 高度なポートフォリオ最適化

今回はまず、`paper simulation` を回すための最小構成を優先する。

---

## 2. テーブル構成

### 2.1 `strategy_param_sets`

役割:
- 銘柄ごとの推奨パラメータセットを保存する
- どの `backtests_runs` を根拠に採用したかを残す
- 将来の複数戦略対応も見越して `strategy_name` を持つ

主なカラム:
- `id`: 主キー
- `strategy_name`: 戦略名。例: `swing_v1`
- `ticker`: 対象銘柄コード
- `param_set_name`: 人が識別しやすい名前。例: `2026-03-17_best`
- `params`: 実際のパラメータ JSON
- `source_run_id`: 根拠となった `backtests_runs.id`
- `is_active`: 現在採用中かどうか
- `effective_from`, `effective_to`: 適用期間
- `notes`: 補足メモ

使い方:
- ある時点で有効な `ticker + strategy_name` の組み合わせから、日次シミュレーションが採用パラメータを読む

### 2.2 `live_positions`

役割:
- 現在保有中、または過去に保有していた建玉を管理する
- どの推奨パラメータセットで建てたかを追えるようにする
- エントリー価格、数量、現在 stop、状態などを持つ

主なカラム:
- `id`: 主キー
- `simulation_name`: シミュレーション名。例: `paper_main`
- `ticker`: 対象銘柄コード
- `strategy_name`: 戦略名
- `param_set_id`: 使用した推奨パラメータセット
- `status`: `OPEN` / `CLOSED`
- `opened_at`, `closed_at`: 建玉開始・終了時刻
- `entry_price`, `exit_price`: 建値・決済値
- `qty`: 数量
- `lot_size`: その時点の売買単位
- `entry_signal_ts`: エントリー根拠シグナル時刻
- `current_stop_price`: 現在有効な stop
- `take_profit_price`: 現在有効な利確目安
- `break_even_armed`, `trailing_armed`: 状態フラグ
- `highest_price_since_entry`: エントリー後高値
- `close_reason`: `TP`, `SL`, `TIME`, `REV`, `FORCE`, `EARNINGS`, `MANUAL` など
- `realized_pnl`: 建玉クローズ時の確定損益
- `fees`: 手数料等
- `meta`: 補助情報 JSON

使い方:
- 日次シミュレーション時に `OPEN` の建玉を読み、更新・決済判定を行う

### 2.3 `live_trade_fills`

役割:
- 実際に発生した売買を約定単位で残す
- `live_positions` との関係を持ち、損益計算や監査に使う

主なカラム:
- `id`: 主キー
- `position_id`: 紐づく建玉
- `simulation_name`: シミュレーション名
- `ticker`: 対象銘柄コード
- `strategy_name`: 戦略名
- `param_set_id`: 使用した推奨パラメータセット
- `executed_at`: 約定時刻
- `side`: `BUY` / `SELL`
- `price`: 約定価格
- `qty`: 数量
- `notional`: 約定代金
- `fee`: 手数料
- `reason`: `ENTRY`, `TP`, `SL`, `TIME`, `REV`, `FORCE` など
- `signal_ts`: 参照したシグナル時刻
- `meta`: 補助情報 JSON

使い方:
- 1 建玉に対して最低 2 レコード（新規買い・決済売り）が入る想定
- 将来、部分決済や買い増しに対応しやすい形にしている

## 3. 関連テーブルとの関係

### 3.1 既存テーブルとのつながり

- `backtests_runs`
  - `strategy_param_sets.source_run_id` で根拠 run を参照する
- `tickers`
  - `strategy_param_sets.ticker`, `live_positions.ticker`, `live_trade_fills.ticker` の参照先になる
- `signals`
  - 直接 FK は貼らないが、`signal_ts` でひもづけられるようにする

### 3.2 今後追加したいもの

- `tickers.lot_size`
- 決算日や権利確定日を持つイベント系テーブル
- 日次シミュレーション結果のサマリー表

---

## 4. 実装時の注意点

### 4.1 `params` は JSONB のまま持つ

理由:
- 現在の `backtests_runs.params` と整合が取りやすい
- パラメータが増減してもスキーマ変更を減らせる

### 4.2 `lot_size` は建玉・約定側にもコピーする

理由:
- 後から `tickers.lot_size` が変わっても、当時の条件を再現しやすい

### 4.3 `simulation_name` を持つ

理由:
- `paper_main`, `paper_alt`, `forward_test_2026q2` のように複数シミュレーションを分けられる

### 4.4 `status` と `close_reason` は text + check 制約にする

理由:
- 可読性が高く、Supabase 上でも扱いやすい

## 5. 次の実装ステップ

この設計の次にやることは、以下が自然。

1. `tickers` に `lot_size` を追加する migration を作る
2. `strategy_param_sets`, `live_positions`, `live_trade_fills` の DDL を確定する
3. 推奨パラメータを投入するための初期 SQL またはスクリプトを作る
4. `paper simulation` 最小版の Python スクリプトを作る
