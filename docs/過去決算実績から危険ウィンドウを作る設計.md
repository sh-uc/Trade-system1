# 過去決算実績から危険ウィンドウを作る設計

この文書は、無料で取得できる過去の決算実績を使って、
実運用で使う「危険ウィンドウ」を推定するための設計案をまとめたものです。

目的は、リアルタイムの正式予定日がなくても、
決算期前後を安全側に回避できる仕組みを作ることです。

---

## 1. 基本方針

考え方は次の2段構えにする。

1. 過去決算実績を `actual` イベントとして蓄積する
2. その実績から、将来の「危険ウィンドウ」を `expected` として推定する

つまり、
- 事実として起きた過去の決算日
- 将来に向けた推定危険期間
を分けて扱う。

これにより、正式予定日がなくても
「この銘柄は毎年この時期が危ない」
という回避ロジックを組みやすくなる。

---

## 2. データモデルの考え方

### 2.1 `ticker_events` に保持するもの

今の `ticker_events` をそのまま使い、`event_type` と `meta` で区別する。

保持イメージ:
- `earnings_actual`
  - 過去実績の決算発表日
- `earnings_expected`
  - 推定した次回危険ウィンドウの中心日
- `ex_dividend`
  - 配当権利落ち日

実装上は次のどちらでもよい。

案A:
- `event_type` を増やす
  - `earnings_actual`
  - `earnings_expected`

案B:
- `event_type='earnings'` のまま
- `meta.kind = actual / expected`

おすすめは案A。
理由:
- SQL で扱いやすい
- simulation 側の条件分岐が分かりやすい

---

## 3. 危険ウィンドウの作り方

### 3.1 必要な入力

銘柄ごとに、最低でも過去 2 ～ 3 年分の決算実績日を集める。

例:
- 2023-05-12
- 2024-05-10
- 2025-05-09

### 3.2 推定の基本ルール

まずは単純で安全側のルールにする。

1. 同じ決算期の過去実績日を集める
2. 日付の中心を求める
3. そこに前後のバッファを付ける

たとえば:
- 過去3年の5月決算発表実績が `05/09`, `05/10`, `05/12`
- 推定中心日を `05/10` とする
- 危険ウィンドウを `05/07 - 05/13` とする

このとき `ticker_events` には、たとえば
- `event_type='earnings_expected'`
- `event_date='2026-05-10'`
- `meta.window_start='2026-05-07'`
- `meta.window_end='2026-05-13'`
のように入れる。

### 3.3 バッファ幅

初版は固定でよい。

おすすめ:
- 前 `3` 営業日
- 後 `1` 営業日

理由:
- 新規停止は少し早めに入れたい
- 強制決済は前日引けまでに済ませたい
- 実運用では安全側に倒す方がよい

---

## 4. 推定ロジックの段階案

### 段階1. 固定月・固定日ベース

- 過去実績日から月日を集計
- 最頻値または中央値を次回の中心日にする
- 固定幅の危険ウィンドウを置く

利点:
- 実装が簡単
- まず動かすには十分

### 段階2. 決算期ごとのクラスタ分け

- 年4回決算の会社では、実績日を四半期ごとに分ける
- それぞれ独立に推定する

利点:
- 5月・8月・11月・2月のようなパターンに対応できる

### 段階3. ばらつき幅を使う

- 過去実績の最小日・最大日を見て、ばらつきが大きい銘柄は窓を広げる

利点:
- 安全側に寄せやすい

おすすめは、まず 段階1 + 2 まで。

---

## 5. paper simulation での使い方

最終的には次のように使う。

### 5.1 新規停止

- `earnings_expected` の危険ウィンドウに入ったら新規建てしない

### 5.2 強制決済

- 保有中に `earnings_expected` の危険ウィンドウへ入る場合
- 前営業日の引けで `EARNINGS_EXPECTED` 理由で決済する

### 5.3 正式日がある場合の優先順位

- `earnings_actual` / 正式予定日がある場合はそちら優先
- なければ `earnings_expected` を使う

つまり優先順位はこうする。

1. 正式予定日
2. 過去実績からの推定危険ウィンドウ
3. 何もない場合は無視

## 6. 追加したい管理項目

将来のために、`meta` には次を持てるようにしておくとよい。

- `window_start`
- `window_end`
- `sample_years`
- `sample_dates`
- `estimation_method`
- `confidence`

例:

```json
{
  "kind": "expected",
  "window_start": "2026-05-07",
  "window_end": "2026-05-13",
  "sample_years": [2023, 2024, 2025],
  "sample_dates": ["2023-05-12", "2024-05-10", "2025-05-09"],
  "estimation_method": "median_mmdd",
  "confidence": "medium"
}
```

---

## 7. 実装ステップ

おすすめの順番はこれです。

1. `ticker_events` に `earnings_actual` / `earnings_expected` を入れられるようにする
2. 過去決算実績を投入するための入口を作る
3. 実績から `earnings_expected` を生成するスクリプトを作る
4. `paper simulation` で `earnings_expected` を参照する

---

## 8. 直近の最小実装案

最初の版はこれで十分。

- 対象銘柄だけ手動またはCSVで過去決算実績を入れる
- `build_earnings_windows.py` のようなスクリプトで、翌年分の危険ウィンドウを生成する
- `simulate_live.py` では `earnings_expected` を見て新規停止 / 強制決済する

このやり方なら、
J-Quants や EDINET の無料データを後から使う場合も、
まずは手元の少数銘柄から安全に育てていける。

---

## 9. 現時点の推奨結論

無料で使える過去実績データだけでも、
決算回避ロジックの土台としては十分価値がある。

特に日本株では、
- 決算期が大きくずれにくい
- 正式予定日がなくても危険月はかなり読める

ため、
`過去実績 -> 推定危険ウィンドウ -> 回避`
の流れは、コストと実用性のバランスが良い。
