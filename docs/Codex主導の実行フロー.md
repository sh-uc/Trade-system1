# Codex主導の実行フロー（初版）

この文書は、`docs/Codex活用指針.md` の「今後やりたいこと」を実際の運用フローに落とした初版です。
目的は、従来の「手順詳細」4～10 を、できるだけ Codex 主導で回せるようにすることです。

---

## 1. 役割分担

### Codex
1. リポジトリ内のコード、SQL、ドキュメントを参照する
2. 必要なコード修正、SQL修正、ドキュメント更新を行う
3. データ取得やバックテスト実行の準備を行う
4. 実行結果を読み取り、要約し、次の改善案を提示する
5. 必要なら commit / push まで行う

### 私
1. その回で検証したいテーマや銘柄群を決める
2. 実行権限や接続先の判断をする
3. Codex の提案を採用するかどうか決める
4. 本番反映の最終判断をする

## 3. 実行フロー

### Step 1. テーマを決める
私が次のような依頼をする。

- 例1: `3778.T のパラメータ改善を進めたい`
- 例2: `東証プライムの候補銘柄を増やしたい`
- 例3: `RSI と MACD 条件の見直し効果を検証したい`

### Step 2. Codex が計画を立てる
Codex はリポジトリと既存結果を確認し、今回の検証で何を変えるかを整理する。

- 対象銘柄
- 変更するパラメータ
- 実行スクリプト
- 保存先テーブル
- 結果確認方法

### Step 3. Codex が必要な修正を行う
必要に応じて、以下を更新する。

- Pythonコード
- SQL
- DDL / migration
- 実行用コマンド
- ドキュメント

### Step 4. Codex が実行する
Codex は必要なコマンドを実行する。

主な対象:
- `yfinance` からのデータ取得
- Supabase への保存
- バックテスト
- 集計SQLの実行

### Step 5. Codex が結果を読む
Codex は実行結果や抽出結果を読み、以下を整理する。

- 成績上位の銘柄
- 成績上位のパラメータ
- ドローダウンや取引回数などの特徴
- 今回の改善が効いたかどうか

### Step 6. Codex が次の提案を返す
結果から次のアクション案を出す。

- 続けて試すべきパラメータ
- 除外すべき銘柄
- ロジック修正の候補
- 本番判断に使える候補セット

### Step 7. 私が採否を決める
私が次の方針を決め、必要なら次のループへ進む。

## 8. 実行環境方針（.venv 前提）

今後は、このリポジトリでは `.venv` を標準の Python 実行環境とみなす。

### 方針
1. Python 実行は原則として `.venv\Scripts\python.exe` を使う。
2. `python` や `py` が端末ごとに異なる場所を向く問題を避けるため、実行スクリプトからはフルパスで `.venv` の Python を呼ぶ。
3. Codex が実行するときも、原則として `.venv` 前提のスクリプトを入口にする。
4. `.venv` が存在しない場合だけ、ユーザー環境の Python を使って作成する。

### 期待する構成
- `.venv\Scripts\python.exe`
- `requirements.txt`
- `run_connectivity_check.ps1`
- `run_bt_sweep_inproc.ps1`

### 初回セットアップの考え方
1. `.venv` を作成する
2. `.venv` に依存関係をインストールする
3. 以後の確認やバックテストは `.venv` の Python で実行する

### メリット
1. Codex とユーザーで同じ Python 実行環境を使いやすい
2. `python` / `py` の解決先ブレを減らせる
3. 実行手順をスクリプト化しやすい
4. 将来の自動化で再利用しやすい

### 推奨セットアップ手順
1. `./setup_venv.ps1` を実行して `.venv` を作成する
2. 必要なら `./setup_venv.ps1 -Recreate` で作り直す
3. 疎通確認は `./run_connectivity_check.ps1` を使う
4. バックテストは `./run_bt_sweep_inproc.ps1` を使う

### セットアップスクリプトの役割
- `setup_venv.ps1`: `.venv` 作成、`pip` 更新、`requirements.txt` インストール
- `run_connectivity_check.ps1`: `.venv` 優先で疎通確認
- `run_bt_sweep_inproc.ps1`: `.venv` 優先でバックテスト実行

### 標準テンプレート
- 結果抽出SQL: `docs/backtest_result_template.sql`
- 評価レポート: `docs/Codex評価レポート標準フォーマット.md`

### 確認を減らすための運用ルール
1. Codex は原則として `setup_venv.ps1`, `run_prepare_codex.ps1`, `run_connectivity_check.ps1`, `run_bt_sweep_inproc.ps1` の入口だけを使う
2. 個別の Python コマンド直打ちは、スクリプト化されていない一時調査のときだけにする
3. sweep 実行時の可変要素は、できるだけ `run_bt_sweep_inproc.ps1` の引数で渡す
4. これにより、同じ入口に対する許可が再利用されやすくなる
5. Codex から出す確認文は、今後は日本語ベースで統一する
