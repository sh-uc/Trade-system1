# 次フェーズ優先順位と実装タスク（2026-03-17）

この文書は、ここまでのバックテスト結果を踏まえて、次に何を優先して進めるべきかを整理したものです。
目的は、アイデアを広げすぎず、実装と検証の順番を決めることです。

---

## 1. 現時点の前提

ここまでの検証で、次のことが分かっている。

- 有力銘柄はすでに複数見つかっている
  - 例: `3103.T`, `7013.T`, `6235.T`, `4506.T`, `6330.T`, `6366.T`
- `BREAK_EVEN_R=0.5`, `TRAILING_START_R=0.5`, `TRAILING_STOP_R=0.25` は有効な標準条件として使える
- 多くの銘柄で `RISK_PCT` と `MAX_HOLD_DAYS` は効きが弱い
- 多くの銘柄で実質的に効いているのは `MACD_ATR_K`
- つまり、売買ロジックは「候補探索と比較」ができる段階に来ている

このため、次の段階では「説明変数を増やしすぎる」より、「運用可能な形に落とす」ことを優先する。

---

## 2. 推奨優先順位

### 優先1. 実運用シミュレーション基盤を作る

最優先にする理由:
- いま見つかっている有力条件が、実際の運用でも耐えるかを確認したい
- `backtest` の数値が良くても、複数銘柄同時保有、資金拘束、建玉管理を入れると結果が変わる
- ここを先に作ると、その後のロジック改善や DeepResearch の目的が明確になる

やること:
- 建玉一覧を持つ
- 銘柄ごとの推奨パラメータセットを保持する
- 日次でシグナル判定する
- 新規建て、保有継続、決済を更新する
- 実現損益、評価損益、拘束資金を記録する

### 優先2. 実運用に必要な制約を入れる

優先する理由:
- これらは後付けすると成績が大きく変わる
- 本番で必要な条件を無視したまま最適化すると、後でやり直しになりやすい

主な対象:
- `LOT_SIZE` の正式対応
- 決算前後の新規停止 / 強制決済
- 権利確定日回避
- 異常時の全建玉クローズ / 新規停止
- 銘柄ごとの売買可否フラグ

### 優先3. 現行ロジックの小さな強化

優先3に置く理由:
- 今のロジックでも比較・選別は十分に機能している
- ここで要素を増やしすぎると、何が効いたか分かりにくくなる

方針:
- 一度に1要素だけ追加する
- まずは既存ロジックの延長線上にあるものから試す

候補:
- trailing の微調整
- イベント前クローズ
- 市場地合いフィルタ
- 銘柄別 `MACD_ATR_K` の扱い

### 優先4. 有力候補銘柄の DeepResearch

優先4に置く理由:
- 調べてから考えるより、回してから調べる方が効率が良い
- 先に運用シミュレーションを回すと、どの特性を調べるべきかが明確になる

調べたいことの例:
- 決算前後の値動き癖
- 為替感応度
- 素材・医薬・機械など業種ごとの反応差
- ギャップ発生頻度
- 出来高急増時の継続率

### 優先5. 外部指標の組み込み

優先5に置く理由:
- 米ドル相場や経済イベントは有望だが、説明変数が一気に増える
- まず個別銘柄ロジックと運用基盤を固めた方が、評価しやすい

候補:
- USD/JPY
- 金利
- 雇用統計やCPIのイベントフラグ
- 業種指数
- 日経平均 / TOPIX レジーム

## 3. 追加でおすすめする案

### 3.1 ポートフォリオ単位の検証

単銘柄ではなく、複数銘柄を同時に持つ前提でシミュレーションする。

理由:
- 実運用では複数銘柄を同時に持つ可能性が高い
- 同時保有数、資金配分、相関の影響を見られる

### 3.2 レジーム判定

市場全体が弱いときは新規停止、強いときだけ新規許可する。

理由:
- 外部指標を大きく増やさずに、地合いの影響を取り込める
- 追加実装の割に効果が出る可能性がある

## 4. 実装ロードマップ

### フェーズ1. 実運用シミュレーション基盤

目標:
- 数ヶ月の疑似運用を日次で回せる状態にする

実装タスク:
1. 推奨パラメータ管理テーブルまたは設定ファイルを作る
2. 建玉管理用テーブルを作る
3. 約定履歴テーブルを作る
4. 日次シミュレーション実行スクリプトを作る
5. 評価損益・資金拘束の集計処理を作る
6. 日次レポート出力を作る

触る可能性が高いファイル / 領域:
- `bt_core.py`
- `db_utils.py`
- 新規 `simulate_live.py` または同等スクリプト
- `supabase/ddl/*`
- `docs/*`

完了条件:
- 有力銘柄セットで 1 日ずつ疑似運用を回せる
- 建玉、実現損益、評価損益が追える

### フェーズ2. 実運用制約の導入

目標:
- 本番で必要な最低限の制約を先に組み込む

実装タスク:
1. `tickers` に `lot_size` を追加する
2. 決算日情報の格納方法を決める
3. 決算前後の取引停止ロジックを追加する
4. 権利確定日回避フラグを追加する
5. 異常時停止フラグを追加する
6. 強制クローズ関数を追加する

触る可能性が高いファイル / 領域:
- `supabase/ddl/030_tickers.sql`
- migration 新規追加
- `db_utils.py`
- シミュレーションスクリプト
- `docs/local_validation_db_setup.md`

完了条件:
- 売買単位が正式反映される
- 重要イベント前後の停止が反映される
- 強制停止の動作確認ができる

### フェーズ3. 小さなロジック強化

目標:
- 1 回に 1 要素ずつ改善して効果を測る

実装タスク候補:
1. trailing の追加微調整
2. イベント前クローズの最適化
3. レジーム判定フィルタの追加
4. 銘柄群ごとの標準条件分岐

完了条件:
- 変更ごとに比較レポートを作れる
- どの改善が効いたか説明できる

### フェーズ4. DeepResearch / 外部要因

目標:
- 有力銘柄の特性や外部要因を必要に応じて追加する

実装タスク候補:
1. `3103.T`, `7013.T`, `4506.T` の特性調査
2. USD/JPY や指数のデータ取得方法を決める
3. 外部要因をフラグ or フィルタとして導入する

完了条件:
- 調査結果がロジックに反映される理由を説明できる
- 追加説明変数がノイズではないと判断できる

## 6. 現時点の推奨結論

次の最優先は、
`もっと複雑な売買ルールを増やすこと` ではなく、
`今見つかっている有力条件を運用できる形に落とすこと`
である。

そのため、まずは
- 実運用シミュレーション基盤
- 実運用制約の導入
を先に進めるのが最も効果的である。
