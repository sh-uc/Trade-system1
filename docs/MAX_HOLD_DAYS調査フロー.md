# MAX_HOLD_DAYS が効かない理由を調べるフロー

この文書は、`MAX_HOLD_DAYS` を変えても結果差がほとんど出ない理由を調べるための標準フローです。
対象は `backtests_runs` / `backtests_trades` を使うバックテスト結果です。

---

## 1. まず確認する仮説

今回のこれまでの結果から、次の仮説が有力です。

1. `TIME` 決済が発生する前に `TP` または `SL` でほぼ終了している
2. `REV` もあまり発生していない
3. そのため `MAX_HOLD_DAYS=3,5,7` を変えても結果差が出ない
4. 実際の保有日数が短く、1日〜4日に集中している

---

## 2. 調べる順番

### Step 1. SELL reason の内訳を見る
- `TP`, `SL`, `REV`, `TIME` の件数を確認する
- `TIME=0` が続いているなら、`MAX_HOLD_DAYS` は効きにくい

### Step 2. 保有日数を見る
- `BUY -> SELL` の日数差を集計する
- `MAX_HOLD_DAYS` より十分短い日数ばかりなら、時間切れの影響はない

### Step 3. signal_ts -> ts を見る
- `REV` / `TIME` が出ている場合だけ、検知日と執行日の差を確認する
- ここで翌営業日寄りになっているかも確認できる

### Step 4. TP / SL の比率を見る
- `TP` と `SL` の偏りを見る
- `SL` に強く偏る銘柄は「早めに落ちる」構造の可能性がある
- `TP` がある程度出る銘柄は、この実装との相性が良い可能性が高い

---

## 3. いま見えていること

代表 run の確認では、以下が見えている。

- `3778.T`: `SL` のみ、保有日数は主に 1 日、最長 4 日
- `3103.T`: `TP` と `SL` の両方あり、保有日数は主に 1 日、最長 4 日
- `3964.T`: `TP` と `SL` の両方あり、保有日数は主に 1 日、最長 4 日
- `4275.T`: `SL` 優勢、保有日数は主に 1 日、最長 4 日
- `TIME` と `REV` は代表 run では 0 件

このため、現時点では `MAX_HOLD_DAYS=5` に届く前に決済されている可能性が高い。

---

## 4. 次の打ち手

### 打ち手A
`MAX_HOLD_DAYS` をさらに短くして差が出るか試す
- 例: `1,2,3`
- これで差が出るなら、今までの `3,5,7` が長すぎた可能性がある

### 打ち手B
`TAKE_PROFIT_RR` や `RISK_PCT` を変えて、保有日数が伸びるか見る
- これにより、`MAX_HOLD_DAYS` が効く局面を意図的に作れる可能性がある

### 打ち手C
`SELL reason` 比率が良い銘柄に絞って検証を続ける
- `TP` が一定数出る銘柄を優先する
- `SL` 偏重の銘柄は後回しにする
